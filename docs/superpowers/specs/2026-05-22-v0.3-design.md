# Maestro v0.3 Design Spec

**Date:** 2026-05-22  
**Status:** Approved  
**Approach:** Incremental — each feature ships independently in priority order.

---

## Feature 1: Per-tool Profile Overrides

### Problem
Tool_result turns (Read/Grep/LS/Glob) represent 40-60% of all turns in a coding session. They currently fall through to the heuristic/embedding/LLM pipeline and land on `standard` (Sonnet). Routing them to Haiku is the highest single savings lever in v0.3.

### Architecture

**New module:** `src/classifiers/tool-override.ts`

Inserts between `turn-type` and `heuristic` in the pipeline. Only fires when `turn-type` has set `turnType = tool_result`. Returns `null` (pipeline fallback) for all other turn types.

**State tracking in `src/wrapper/sdk-proxy.ts`:**

Maintain a bounded `Map<string, string>` (`tool_use_id → tool_name`, max 50 entries). Populate it by scanning each assistant message for `tool_use` content blocks before forwarding. When a `tool_result` turn arrives, look up the tool name by `tool_use_id`.

**Routing table:**

| Tool(s) | Class | Confidence |
|---|---|---|
| Read, Glob | trivial | 1.0 |
| Grep, LS | trivial | 1.0 |
| Edit, Write, MultiEdit | simple | 1.0 |
| Bash | simple | 1.0 |
| Task, WebFetch, WebSearch | standard | 1.0 |
| Unknown / MCP / unresolved id | pipeline fallback | — |

**Fallback:** when tool name cannot be resolved, return `null` — pipeline continues normally.

### Files changed
- `src/classifiers/tool-override.ts` (new)
- `src/classifiers/tool-override.test.ts` (new)
- `src/wrapper/sdk-proxy.ts` (add tool_use_id tracking)
- `src/cli/wire-compat.ts` (insert tool-override into pipeline)

### Testing
- Unit: tool name → class mapping for all table entries
- Unit: unknown tool_use_id → null (pipeline fallback)
- Integration: sdk-proxy emits set_model for tool_result turns based on tracked tool name

## Feature 2: Tournament Matrix (model × effort)

### Problem
Current tournament only tests model-tier downgrade (e.g., Opus → Sonnet). Effort reduction within the same model (Sonnet/high → Sonnet/low) also saves money but is never tested.

### Architecture

Extend `src/eval/tournament.ts` with a `--tournament-matrix` boolean flag (default off, backward compatible).

When `--tournament-matrix` is set:
- For each sampled prompt, test the full grid: current tier + all cheaper model tiers + same model at lower effort levels
- Judge runs once per unique `(model, effort)` pair
- Per-class aggregation tracks win rate per cell
- `--tournament-output` writes both model downgrades and effort downgrades to the proposal

Cost estimation runs upfront and requires `--confirm-cost` if estimate > $1 (existing gate).

### Files changed
- `src/eval/tournament.ts` (add matrix mode)
- `src/eval/tournament.test.ts` (new matrix test cases)

## Feature 3: Per-project Config

### Problem
Different repos have different complexity profiles. Today all projects share a single `~/.maestro/config.json`. A scripts repo shouldn't use the same budget as a distributed systems codebase.

### Architecture

**New module:** `src/core/config-discovery.ts`

On each wrapper invocation, walks up from `process.cwd()` looking for `.maestro/config.json`. Stops at home directory or filesystem root. Result is cached for the process lifetime (one invocation).

**Merge order** (lowest → highest priority):
```
built-in defaults
  → user config (~/.maestro/config.json)
    → project config (<cwd>/.maestro/config.json)
```

**Allowed fields in project config** (subset of user config):
- `profile` — override the active profile name
- `profileOverrides` — per-class spec overrides
- `excludeDynamicSections` — per-project toggle
- `useEmbeddingClassifier` — opt out for repos where it's not helping

**Not allowed in project config:** `telemetryPath`, `feedbackSampleRate`, `useLlmClassifierInWrapper` (user-level concerns only).

No new CLI commands. Discovery runs automatically in the wrapper hot path.

### Files changed
- `src/core/config-discovery.ts` (new)
- `src/core/config-discovery.test.ts` (new)
- `src/cli/utils.ts` (integrate discovery into `loadCliConfig`)

## Feature 4: `maestro init` / `maestro doctor`

### Problem
New-machine setup requires running `install.sh` or following manual README steps. There's no way to verify the environment is correctly configured without checking each piece manually.

### Architecture

**`maestro init`** (`src/cli/init.ts`):
- Interactive setup wizard, idempotent (safe to re-run)
- Steps: write `~/.maestro/config.json` defaults, call `install-vscode`, install slash commands, install stop-feedback hook
- Replaces the manual post-install checklist from `maestro guide`
- Prints a summary of what was done / what was already in place

**`maestro doctor`** (`src/cli/doctor.ts`):
- Non-destructive environment check, no side effects
- Checks: claude binary present + version in range, `claudeProcessWrapper` set in VSCode settings, `~/.maestro/config.json` exists, telemetry file writable, Node ≥ 20, pnpm available
- Prints pass/fail list with one-line fix instructions per failure
- Exit code 0 if all pass, 1 if any fail (CI-friendly)

### Files changed
- `src/cli/init.ts` (new)
- `src/cli/doctor.ts` (new)
- `src/cli/index.ts` (register commands)

## Feature 5: `--fast-mode` Cost Profile (spike-gated)

### Problem
`claude --output-format json` output includes `"fast_mode_state"`. Anthropic's fast mode may have a cost dimension that Maestro could exploit per class. Unknown until spiked.

### Spike
Extend `wrapper/preflight.ts` to capture `fast_mode_state` from the preflight invocation output. Run a controlled test: same prompt, fast mode on vs off, compare `total_cost_usd` and `duration_ms`. Document findings in `docs/router-observations.md`.

**Gate:** no code ships until the spike confirms a cost benefit. If fast mode is speed-only, close the feature and update `docs/future-ideas.md`.

**If cost benefit confirmed:**
- Add `fastMode?: boolean` to `ClassSpec` in `core/types.ts`
- Expose per-class in profile definitions
- Add a `fast` profile alongside `balanced` and `cheap`

### Files changed (if spike confirms benefit)
- `src/core/types.ts` (add `fastMode` to `ClassSpec`)
- `src/core/profile.ts` (add `fast` profile)
- `src/wrapper/spawn.ts` (pass fast mode flag when set)
- `wrapper/preflight.ts` (capture fast_mode_state)

## Implementation Order

1. Per-tool override (ships as `v0.3.0` — highest daily savings impact)
2. Tournament matrix (ships as `v0.3.1`)
3. Per-project config (ships as `v0.3.2`)
4. `maestro init` / `maestro doctor` (ships as `v0.3.3`)
5. Fast-mode spike → implement if confirmed (ships as `v0.3.4` if warranted)

## Model Routing for Implementation

| Feature | Class | Model |
|---|---|---|
| Per-tool override | hard | Sonnet/high |
| Tournament matrix | standard | Sonnet/medium |
| Per-project config | simple | Sonnet/low |
| maestro init/doctor | simple | Sonnet/low |
| Fast-mode spike | trivial | Haiku |
